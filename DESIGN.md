# 자동매매 시스템 설계 (v0.1 — 초안)

> 대상: **암호화폐 레버리지(무기한 선물, Perpetual Futures)**
> 핵심 컨셉: 유저가 **매매기법을 프리셋(블록 조립)으로 만들어** 백테스트 → 모의 → 실거래로 굴린다.

---

## 0. 설계 대전제

레버리지 매매는 현물과 근본적으로 다르다. 아래 3개를 데이터 모델의 1급 시민으로 넣는다.

1. **청산(Liquidation)** — 방향 틀리면 증거금 소진 → 강제 청산. 손절보다 먼저 터질 수 있다.
2. **증거금/레버리지(Margin/Leverage)** — 포지션 크기 = 증거금 × 레버리지. 사이징이 곧 생존.
3. **펀딩비(Funding Rate)** — 8시간마다 롱/숏이 서로 지불. 장기 보유 시 수익률을 갉아먹는다.

> "백테스트에서 청산·펀딩비를 반영 안 하면 그건 백테스트가 아니라 희망회로다."

---

## 1. 시스템 구조 (레이어)

```
┌─────────────────────────────────────────────┐
│  1. 프리셋 빌더 (UI 블록 조립)                 │  ← 유저가 매매기법 정의
├─────────────────────────────────────────────┤
│  2. 전략 엔진 (Strategy Engine)               │  ← 프리셋 → 신호(signal) 판정
├─────────────────────────────────────────────┤
│  3. 실행 레이어 (Execution)                    │  ← 신호 → 주문. 3모드 공유
│     ├ 백테스트  (과거 데이터)                   │
│     ├ 페이퍼    (실시간 시세 + 가짜 주문)        │
│     └ 실거래    (거래소 API)                    │
├─────────────────────────────────────────────┤
│  4. 리스크 가드레일 (계정 전역 안전장치)         │  ← 프리셋과 무관하게 강제
├─────────────────────────────────────────────┤
│  5. 데이터/알림/로그                            │  ← 시세 수집, 기록, 알림
└─────────────────────────────────────────────┘
```

핵심 설계 원칙: **같은 전략 엔진을 백테스트·페이퍼·실거래가 공유한다.**
→ 백테스트에서 통과한 로직이 실거래에서 "다르게" 동작하면 안 됨. 실행 어댑터만 교체.

---

## 2. 프리셋 = 매매기법 (블록 조립)

프리셋은 결국 아래 4개 블록 묶음의 조합이다.

```
프리셋
├─ 진입(Entry)     : 언제 포지션을 여는가 (조건 트리)
├─ 청산(Exit)      : 언제 닫는가 (익절/손절/조건/시간)
├─ 사이징(Sizing)  : 얼마나 크게 (레버리지·증거금 비중)
└─ 필터(Filter)    : 언제 아예 매매 안 하는가 (변동성/시간대/펀딩비)
```

### 블록 카탈로그 (유저가 UI에서 끌어 쓰는 재료)

**지표 블록 (Indicator)**
- 이동평균(SMA/EMA), RSI, MACD, 볼린저밴드, ATR, 거래량, 스토캐스틱 …
- 각 블록은 파라미터를 가짐 (예: RSI 기간=14)

**조건 블록 (Condition)**
- 비교: `RSI < 30`, `가격 > EMA(20)`
- 교차: `MACD 골든크로스`, `가격이 EMA 상향 돌파`
- 조합: `AND / OR / NOT` 로 트리 구성

**액션 블록 (Action)**
- 롱 진입 / 숏 진입 / 청산 / 부분청산 / 레버리지 조정

**리스크 블록 (Risk) — 레버리지 필수**
- 손절 `-X%` / 익절 `+Y%`
- 트레일링 스탑 (고점 대비 -Z%)
- 최대 레버리지 상한
- 청산가로부터 최소 안전마진 (예: 청산가까지 15% 이상 여유 없으면 진입 금지)

### 프리셋 저장 형태 (내부 표현)

UI 블록은 결국 이런 **JSON 트리**로 직렬화된다. UI는 껍데기, 진실은 이 스키마.

```jsonc
{
  "name": "RSI 과매도 반등 롱",
  "symbol": "BTCUSDT",
  "timeframe": "15m",
  "entry": {
    "op": "AND",
    "children": [
      { "indicator": "RSI", "period": 14, "cmp": "<", "value": 30 },
      { "indicator": "price", "cmp": ">", "ref": { "indicator": "EMA", "period": 200 } }
    ]
  },
  "exit": {
    "takeProfit": { "type": "percent", "value": 3.0 },
    "stopLoss":   { "type": "percent", "value": 1.5 },
    "trailing":   { "enabled": true, "callback": 0.8 }
  },
  "sizing": {
    "leverage": 5,
    "marginPercent": 10,          // 계정 증거금의 10%를 이 포지션에
    "minLiquidationBuffer": 15    // 청산가까지 15% 여유 없으면 진입 안 함
  },
  "filter": {
    "maxSpread": 0.05,
    "avoidFundingWindow": true,   // 펀딩 정산 직전 진입 회피
    "tradingHours": "24h"
  }
}
```

> 이 스키마가 있으면: 블록 UI ↔ JSON ↔ 백테스트 엔진이 전부 같은 언어를 쓴다.
> 유저 공유(프리셋 마켓), 버전관리, 복제-수정이 전부 공짜로 따라온다.

## 3. 실행 3모드 (검증 파이프라인)

```
[백테스트] ──통과──> [페이퍼(모의)] ──통과──> [실거래]
 과거 데이터        실시간 시세+가짜주문      진짜 돈
```

- **백테스트**: 과거 캔들로 성과 측정. 청산·펀딩비·수수료·슬리피지 반드시 반영.
- **페이퍼**: 실시간 시세로 며칠~몇 주 돌려봄. "백테스트는 좋은데 실시간엔 신호가 안 뜨는" 함정 검출.
- **실거래**: 검증된 프리셋만 승격. 처음엔 소액/저레버리지 강제.

성과 지표(프리셋에 항상 붙어다님):
- 총수익률 / **MDD(최대낙폭)** / 승률 / 손익비 / 샤프 / **청산 횟수** / 펀딩비 누적

---

## 4. 리스크 가드레일 (계정 전역 — 가장 중요)

프리셋이 뭐라고 하든 **계정 레벨에서 강제되는 최후의 방어선**. 버그·폭락·API장애로부터 계좌를 지킴.

- **일일 손실 한도**: 하루 -X% 도달 시 당일 전 매매 정지 (kill switch)
- **최대 동시 포지션 수 / 총 노출 한도**: 레버리지 합산 상한
- **연속 손실 차단**: N연속 손절 시 쿨다운
- **API/네트워크 장애**: 시세 끊기거나 주문 실패 반복 시 신규 진입 중단, 알림
- **비상 전량청산 버튼**: 사람이 언제든 한 방에 다 닫을 수 있어야 함
- **레버리지 상한**: 유저가 프리셋에 20배 넣어도 계정 정책이 10배로 캡

> 자동매매 사고의 90%는 "전략이 틀려서"가 아니라 "안전장치가 없어서" 터진다.

---

## 5. 데이터 · 알림 · 로그

- **시세 수집**: 거래소 WebSocket(실시간) + REST(과거 캔들). 재연결/결측 처리.
- **로그(감사 추적)**: 모든 신호·주문·체결·에러를 타임스탬프로 기록. "안 볼 때 뭐 했나" 재구성 가능해야.
- **알림**: 텔레그램/슬랙/디스코드로 진입·청산·손절·에러·kill switch 발동 푸시.

## 7. 열린 질문 (다음에 정할 것)

1. **거래소**: 바이낸스 선물? 바이비트? (테스트넷 지원 여부가 중요 — 페이퍼트레이딩용)
2. **블록 세트 범위**: 지표 몇 개부터 시작? (RSI/EMA/MACD 3개로 MVP?)
3. **백테스트 데이터**: 어디서 과거 캔들 확보? (거래소 API vs 데이터 제공처)
4. **UI 우선순위**: 블록 빌더 UI를 먼저? 아니면 JSON으로 프리셋 쓰고 엔진부터?
5. **혼자 쓸 거 vs 나중에 서비스화**: 계정/권한 설계를 지금 넣을지

## 8. 제안하는 다음 단계 (MVP 순서)

> UI 블록 빌더는 매력적이지만 **엔진이 없으면 껍데기**다. 안에서 밖으로 짓자.

1. **프리셋 JSON 스키마 확정** (블록 카탈로그 v1: RSI/EMA/MACD + 손절/익절/레버리지)
2. **백테스트 엔진** — 청산·펀딩비·수수료 반영. 프리셋 하나 넣으면 성과 나옴
3. **페이퍼 트레이딩** — 실시간 시세로 같은 엔진 구동
4. **리스크 가드레일** — kill switch, 손실 한도
5. **실거래 어댑터** (ccxt) — 소액부터
6. **블록 빌더 UI** — 이때쯤 JSON을 유저가 안 봐도 되게
